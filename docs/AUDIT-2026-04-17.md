# Content-to-Code Audit — 2026-04-17

Full audit of research nodes, ability domains, pair synergies, and emergent rules. Goal: every JSON-defined effect must be consumed by TypeScript code, and every code path must have a JSON definition.

---

## 1. Research Nodes (30 nodes, 10 domains)

### Structural Issue

All `qualitativeEffect.effect` key-value pairs (e.g. `poisonExtraStack: 1`, `armorPenetration: 0.5`) and `type` strings in `research.json` are **documentation-only**. No TypeScript code reads them. The `resolveResearchDoctrine()` function maps node completion via `hasNode()` to boolean flags, and game systems hardcode the actual numeric values. Only the `description` field is consumed (by the UI).

This means the JSON values can drift from the hardcoded values without any test catching it. Long-term fix: code should read from JSON. Short-term: at minimum ensure descriptions match what the code actually does.

### Dead Doctrine Flags

These flags are set in `capabilityDoctrine.ts` but never read by any system:

| # | Flag | Node | Claimed Effect |
|---|------|------|----------------|
| R1 | `forcedMarchEnabled` | charge_t1 | No cooldown on first charge |
| R2 | `poisonBonusEnabled` | venom_t3 (foreign) | +50% poison damage |
| R3 | `toxicBulwarkEnabled` | venom_t3 (native) | All units apply poison on hit |
| R4 | `permanentStealthEnabled` | camel_adaptation_t2 | Permanent stealth in desert |

### Phantom Flag

| # | Flag | Source | Issue |
|---|------|--------|-------|
| R5 | `zoCAuraEnabled` | fortress_t2 | Consumed by `zocSystem.ts` but fortress_t2 JSON has no ZoC effect defined. The JSON now says "build field fortifications" but the code also grants ZoC projection — an undocumented side effect. |

### Valid References

All 10 `codifies` fields (T1 nodes) correctly reference domain IDs in `ability-domains.json`. All `unlocks` arrays are empty (no broken prototype references).

## 2. Ability Domains (10 domains)

**Status: All LIVE.** No dead domains.

All 10 domains (`venom`, `fortress`, `charge`, `hitrun`, `tidal_warfare`, `nature_healing`, `camel_adaptation`, `river_stealth`, `slaving`, `heavy_hitter`) are wired into civilizations, research, synergies, and emergent rules.

Minor: `baseEffect.type` and `baseEffect.effect` discriminator strings are never switched on by code. They serve as documentation only.

### Capability Domains

`capability-domains.json` defines 13 domains. The registry API (`getCapabilityDomain`, `getAllCapabilityDomains`) is defined but never called — the domain IDs are only used as string keys in `capabilityPressure` maps on terrain/chassis/components. Three domains (`shock_resistance`, `desert_survival`, `endurance`) accumulate only passively from terrain exposure and have no combat signal wiring.

- **High value** (core combat loops): S1 `lethal_ambush`, S6 `ambush_charge`, S9 `formation_crush`, S18 `armor_shred`
- **Medium value** (capture/territory): S2 `poison_capture`, S7 `capture_charge`, S10 `capture_retreat`, S13 `naval_capture`, S17 `stealth_capture`
- **Lower value** (stat modifiers): S3-S5, S8, S11, S12, S14-S16, S19-S23
